What is a lineage? (In your terms: what does a goal_lineage vs resolution_lineage identify, and why parent pointers + expr* / rule* instead of storing the full path?)

So a lineage is a derivation path. If you want to uniquely identify a goal or uniquely identify a SPECIFIC resolution taking place (like X goal resolved against Y rule), that is the system that handles that. The reason they are called lineages is because each lineage object (goal_lineage for example) stores enough information to be able to retrace how that goal was derived since the start of the sim. It is literally the lineage of the goal (which includes prior goals and prior resolutions). Each lineage object can be thought of as a linked list (is a linked list) with the terminating node being nullptr (the root of the tree).



Each type of lineage has a predecessor which is of the opposite type of lineage. so it goes resolution -> goal ->resolution -> goal -> ...

The reason for this is to accurately model the behavior of the CHC search as what it is: an AND-OR tree. For each set of goals, we want ALL of them to be resolved, and for each goal, we have several CHOICES (or) for how they can be resolved.



To answer your earlier q: Lineages are referenced in decision lemmas which are learned by CDCL between sims. This is when we pin the lineages, since we are going to need those lineages to be accessible for future sims in the CDCL object.



Also, lineage_pool is the current way of making lineages, since it is a highly memory-efficient way to represent many lineages. They have cons-hashing so two lineages that share a parent ACTUALLY share that same parent (in memory). Making it so we end up with effectively a tree of lineages in the lineage_pool.

Clear picture: lineages are cons-hashed derivation paths in an AND-OR tree; pinned when embedded in CDCL lemmas across sims.

Next: What does MHU do during sim, in one sentence — propagate failed unifications, detect equalities, or something else?

propagate failed unifications. So basically, there is this problem which is we have a bunch of "uncommitted" unifications of active goals to active candidate heads (copies of db rule heads)



We need to then ACCEPT (commit to) one of these prospective unifications (which accepts the new bindings from that bind_map into the common bind_map). However, when doing this, the previously existing unify_heads (those which still exist and were not chosen to commit yet) need to have their local bind maps REBASED on the new common bind_map, WHICH MAY CAUSE THEM TO NOW BE CONFLICTING. (colliding unification). When this happens, it means that that candidate has been ruled out.

So MHU maintains prospective unifications and eliminates candidates whose heads become incompatible after another commit rebases the common bindings.

Next: What triggers committing one unify-head into common_bind_map — a resolver::resolve, a decision resolution, or something dedicated?

I won't say any specific part of the code since that may change. But generally, it happens BEFORE the process of activating new candidates during the resolution step.



so it goes:



Eliminate old candidates -> Try to activate new candidates



The reason for this order is: The elimination backlog. If we are able to determine that some candidates should be eliminated BEFORE they are created, we add the eliminations to the backlog, and then BEFORE creating (activating) those candidates, we just check if they are in the backlog and avoid making them altogether.

Eliminate-then-activate ordering + backlog avoids creating candidates you already know are dead.

Next: During sim, what does CDCL constrain do — only enforce learned decision-set avoidances, or also derive new eliminations mid-run?

It derives new eliminations when any avoidance drops down to unit (only having one member). Because a multi-entry avoidance means "avoid doing these several things simultaneously". One thing means avoid doing this one thing at all costs. When this occurs, we return them as eliminations (really, we yield them via a coroutine yield). The reason we use a coroutine for yielding eliminations is, we want to detect conflicts in the sim early (when any goal has zero candidates). We may not need to know about all of the eliminations to be able to find our first conflict, and terminate the sim early.

Unit avoidances become immediate eliminations; coroutine yields let sim stop at the first zero-candidate conflict without draining all propagations.

Next: What does normalizer / whnf do in your engine — full term normalization, or mainly follow variable bindings?

whnf() is a function belonging to i_bind_map. And it serves the sole purpose of implementing a collapsing union-find in the map of var bindings. As weak head normal form states, it only looks up the union find in the top-level equivalence class. So if we reach a functor at the end of the equivalence class, but it has some variable children, we don't recur to the children. The functor is a base case for whnf. Really, functor is a base case for whnf just because any terminal entry in an equivalence class is the base case. And functors themselves, when belonging to an equivalence class ARE the terminal entity, tautologically.

Union-find collapse at the top level only — functors are WHNF terminals, no recursive normalization under them.
